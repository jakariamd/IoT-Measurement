</li></ul></div> <div><h1>  Privacy best practices</h1><span>  bookmark_border</span><span>  
          
          Stay organized with collections
        </span><span>  
          
          Save and categorize content based on your preferences.
        </span><p>  Android is focused on helping users take advantage of the latest innovations
while making their security and privacy top priorities.</p><p>  Some of the best practices described on this page also appear in the
<a href="/privacy/cheat-sheet" target="_blank"> cheat sheet </a>.</p></div> <div><h2>  Pay attention to permissions</h2><p>  
Build trust with your users by being transparent and providing users control
over how they experience your app.</p><ul>  <li><a href="/training/permissions/usage-notes#avoid_requesting_unnecessary_permissions" target="_blank"> Request the minimum permissions that your feature
needs </a>.
Whenever you introduce major changes to your app, review the requested
permissions to confirm that your app's features still need them.
</li><li>Keep in mind that newer versions of Android often introduce ways to
access data in a privacy-conscious manner without requiring permissions.
For more information, see <a href="/training/permissions/evaluating" target="_blank"> Evaluate whether your app needs to declare permissions </a>.</li></ul></div> <ul><ul>  <li>If your app is distributed on Google Play, <a href="/topic/performance/vitals/permissions#use_android_vitals_to_gauge_user_perceptions" target="_blank"> Android
vitals </a>
tells you the percentage of users who deny permissions in your app. Use
this data to reassess the design of features whose required permissions
are most commonly denied.</li><li>Follow the recommended flow to <a href="/training/permissions/requesting#explain" target="_blank"> explain why a feature in your app needs a
permission </a>. Request the permission
when it's needed, rather than at app startup, so that the permission need is
clear to users.</li></ul></ul> <ul><ul>  <li>Keep in mind that users or the system can deny the permission multiple times.
Android respects this user choice by ignoring permission requests from the same
app.</li><li>Gracefully degrade when users deny or revoke a permission. For example, you
can disable your app's voice input feature if the user doesn't grant the
microphone permission.</li><li>When you update your app,
<a href="/training/permissions/requesting#remove-access" target="_blank"> remove your app's access </a> to
any runtime permissions that your app no longer needs.</li></ul></ul> <ul><ul>  <li>If you are using an SDK or library that accesses data guarded by dangerous
permissions, users generally attribute this to your app. Make sure you
<a href="/training/permissions/usage-notes#sdk-libraries" target="_blank"> understand the permissions that your SDKs require and
why </a>.
</li><li>If you test your app on Android 11 (API level 30), use <a href="/about/versions/11/privacy/data-access-auditing" target="_blank"> data access
auditing </a> to discover
places in your code and in third-party library code where private data is
being accessed.</li></ul></ul> <div><h2>  Minimize your use of location</h2><p>  If your app can support its use cases without requiring any location data,
don't request any location permissions. If your app requests permission to
access location, help users make an informed decision.</p><ul>  <li>If your app needs to collect location information, explain to users how your
app uses this information to deliver specific benefits to them.</li><li>If your app needs to pair the user's device with a nearby device over
Bluetooth or Wi-Fi, use the <a href="/guide/topics/connectivity/companion-device-pairing" target="_blank"> companion device
manager </a>, which doesn't
require location permissions.</li><li>Review the <a href="/training/location/retrieve-current#permissions" target="_blank"> level of location
granularity </a> that your app
needs. Coarse location access is sufficient to fulfill most location-related use
cases.</li></ul></div> <ul><ul>  <li>Access location data while your app is visible to the user. That way, users
can better understand why your app requests location information.</li><li>If your app requires background location, such as when implementing
geofencing, make sure that it's critical to the core functionality of the app
and is done in a way that's obvious to users. Learn more about <a href="/training/location/background" target="_blank"> considerations
for using background location </a>.</li><li>On Android 10 (API level 29) and higher, users can limit your app's location
access to while the app is in use. Design your app so that it degrades
gracefully when it doesn't have all-the-time access to location.</li></ul></ul> <ul><ul>  <li>If your app needs to retain location access for a user-initiated ongoing task
after the user navigates away from your app's UI, <a href="/guide/components/services#Foreground" target="_blank"> start a foreground
service </a> before your app goes into the
background. You can do this in one of Android's lifecycle callbacks, such as
.</li><li>Don't initiate foreground services from the background. Instead, consider
launching your app from a notification and then executing the location code when
your app's UI becomes visible.</li></ul></ul> <div><h2>  Handle data safely</h2><p>  
<b> Note:  </b> You can read more about what's considered sensitive data in the
  <a href="https://play.google.com/about/privacy-security-deception/user-data/#!?zippy_activeEl=personal-sensitive#personal-sensitive" target="_blank"> User Data article </a> page in the Google Play Developer Policy Center.
</p><p>  
Be transparent and secure in how you handle sensitive data.</p><ul>  <li>Make users aware of when and why your app collects, uses, or shares sensitive
data.</li><li>Use the scoped storage model if possible. <a href="/training/data-storage/use-cases" target="_blank"> Learn how to migrate to scoped
storage based on your app's use cases </a>.</li><li>Always use <a href="/topic/security/best-practices#network" target="_blank"> secure network
connections </a>. For your app's data at
rest, use Android's built-in credential encryption.
For data in transit, <a href="/training/articles/security-ssl" target="_blank"> use TLS, the successor of SSL, for all data
transmission </a> regardless of sensitivity.</li></ul></div> <ul><ul>  <li>Place files that contain sensitive data in your app-private directory
within <a href="/training/data-storage/files/internal" target="_blank"> internal storage </a>.</li><li>On Android 10 (API level 29), store files only relevant to your app in the
<a href="/training/data-storage/app-specific#external" target="_blank"> app-specific directory in external
storage </a>.
<a href="/training/data-storage#scoped-storage" target="_blank"> Learn more about scoped storage </a>.</li><li>If you need to pass sensitive data to another app, use
<a href="/guide/components/intents-filters#Types" target="_blank"> an explicit intent </a>.
Grant <a href="/topic/security/best-practices#permissions-share-data" target="_blank"> one-time data access </a>
to further restrict the other app's access.</li></ul></ul> <ul><ul>  <li>Even when your app is in the foreground, show a
real-time indication that you are capturing from the microphone or camera. Note
that Android 9 (API level 28) and higher <a href="/about/versions/pie/android-9.0-changes-all#privacy-changes-all" target="_blank"> don't allow for microphone or camera
access </a> when
your app is in the background.</li><li>Don't include sensitive data in logcat messages or your app's log files.
<a href="/training/articles/security-tips#UserData" target="_blank"> Learn more about how to handle user
data </a>.</li></ul></ul> <p><p>  Jetpack offers several libraries to keep your app's data more secure. Learn
more in the guides on using the
<a href="/topic/security/data" target="_blank"> Jetpack Security library </a> and the
<a href="/guide/topics/ui/settings/use-saved-values" target="_blank"> Jetpack Preferences library </a>.</p></p> <div><h2>  Use resettable identifiers</h2><p>  
Respect your users' privacy and use resettable identifiers. See
<a href="/training/articles/user-data-ids" target="_blank"> Best practices for unique identifiers </a>
for more information.</p><ul>  <li>Don't access IMEI or the device serial number, as these identifiers are
persistent. If you try to access these identifiers in an app that targets
Android 10 (API level 29) or higher, a
 occurs.</li><li>Only <a href="/training/articles/user-data-ids#advertising-ids" target="_blank"> use an advertising ID for user profiling or ads use
cases </a>. For apps in Google
Play, this is a requirement. Always respect user preferences on advertisement
tracking for personalization.</li><li>For the vast majority of non-ads use cases, <a href="/training/articles/user-data-ids#signed-out-anon-user-analytics" target="_blank"> use a privately stored
globally-unique ID
(GUID) </a>, which
is app-scoped.</li></ul></div> <ul><ul>  <li>Use the secure settings Android ID (SSAID) to share state between the apps
that you own without requiring the user to sign in to an account. Learn more
about how to <a href="/training/articles/user-data-ids#signed-out-user-prefs-between-apps" target="_blank"> track signed-out user preferences between
apps </a>.</li></ul></ul> <p><p>  Content and code samples on this page are subject to the licenses described in the <a href="/license" target="_blank"> Content License </a>.
